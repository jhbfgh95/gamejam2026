# AGENTS.md

## 프로젝트 개요
- 이 프로젝트는 Unity 2D 클릭/터치 기반 프로젝트입니다.
- 물리 기반 이동이나 충돌보다 UI, 선택, 배치, 상태 변화, 상호작용 로직이 중요합니다.
- 기존 씬, 프리팹, 에셋, 폴더 구조가 있다면 그 방식을 우선 따릅니다.

## 작업 원칙
- 변경은 작고 명확하게 합니다.
- 요청받지 않은 리팩터링이나 구조 변경은 하지 않습니다.
- 사용자가 만든 변경사항을 되돌리지 않습니다.
- 작업과 무관한 파일은 건드리지 않습니다.
- Unity 에셋을 추가, 이동, 이름 변경, 삭제할 때는 관련 `.meta` 파일을 함께 유지합니다.
- 씬, 프리팹, 애니메이터, ScriptableObject 변경은 최소화하고 변경 이유를 설명합니다.

## 코드 규칙
- C# 스크립트는 기본적으로 `Assets/Scripts` 아래에 둡니다.
- 클래스, 메서드, public 프로퍼티는 PascalCase를 사용합니다.
- private 필드와 지역 변수는 camelCase를 사용합니다.
- 인스펙터에서 조정할 값은 `public` 필드보다 `[SerializeField] private` 필드를 선호합니다.
- `MonoBehaviour`는 한 가지 역할에 집중하도록 작게 유지합니다.
- 이미 프로젝트에 있는 네이밍, 폴더 구조, 이벤트 연결 방식을 우선 따릅니다.
- 복잡한 로직은 짧은 메서드로 나누되, 불필요한 추상화는 만들지 않습니다.

## 클릭/터치 상호작용
- 입력 처리는 클릭/터치 중심으로 설계합니다.
- UI 버튼은 `Button.onClick` 또는 프로젝트의 기존 UI 이벤트 패턴을 따릅니다.
- 오브젝트 클릭, 선택, 드래그, 배치 로직은 기존 구현 방식이 있으면 그 방식을 우선 사용합니다.
- 마우스 입력만 가정하지 말고 모바일 터치 가능성도 고려합니다.
- 화면 좌표, 월드 좌표, UI 좌표 변환을 명확히 구분합니다.
- 카메라, 캔버스, 레이캐스트 설정 변경은 필요한 경우에만 수행하고 이유를 설명합니다.
- 불필요하게 `Rigidbody2D`, `Collider2D`, 물리 충돌, 힘 기반 이동 로직을 도입하지 않습니다.

## Unity 에셋 관리
- `Library/`, `Temp/`, `Obj/`, `Build/`, `Builds/`, `Logs/` 폴더는 수정하지 않습니다.
- 자동 생성 파일이나 캐시 파일은 커밋 대상 작업에서 제외합니다.
- 씬 파일(`.unity`)과 프리팹 파일(`.prefab`)은 변경 범위가 커지기 쉬우므로 특히 조심합니다.
- 에셋 참조가 깨지지 않도록 파일 이동이나 이름 변경은 꼭 필요할 때만 합니다.

## 테스트와 확인
- 코드 변경 후 C# 컴파일 오류가 없는지 확인합니다.
- 가능하면 Unity Editor에서 관련 씬을 열어 동작을 확인합니다.
- 클릭/터치 상호작용 변경 시 다음을 확인합니다.
  - 클릭 또는 터치가 의도한 대상에만 반응하는지
  - UI 클릭과 월드 오브젝트 클릭이 충돌하지 않는지
  - 선택, 해제, 취소, 반복 클릭이 자연스럽게 동작하는지
  - 모바일 터치 환경에서도 문제가 없을 가능성이 높은지

## 응답 방식
- 기본적으로 한국어로 답변합니다.
- 설명은 짧고 실용적으로 합니다.
- 변경한 파일과 이유를 간단히 요약합니다.
- 확실하지 않은 내용은 단정하지 않고 확인이 필요하다고 말합니다.
- Unity Editor에서 사용자가 직접 확인해야 하는 내용은 단계별로 안내합니다.
- 전문 용어는 필요할 때만 간단히 풀어서 설명합니다.
